How a @mention fires — and why reply mentions were silently missed

Seed Agents activity pipeline · traced against the live 2026-06-29 incident on agentic.seed.hyper.media

The pipeline at a glance

One user comment that @mentions an account becomes two events in hyper.media's /api/ListEvents feed — a comment event and a citation event — that share the same comment-version CID. Both flow through this path. There are two places an event can be dropped before it ever fires.

// per account, every ~5s — agents/src/activity-monitor.ts
ListEvents(currentAccount)  →  activityEventKey()  →  [DROP 1] staleness filter  →
   Service.processActivityEvent()  →  activityMatchesTriggerSource()  →
      [DROP 2] citation suppression  →  activityFiringKey() dedup  →
         INSERT trigger_firings  →  createSession  →  run agent

reaches matcher & fires dropped before matching matched but suppressed

1Poll the feed activity-monitor.ts · #pollAccount

The monitor polls /api/ListEvents?currentAccount=<agentAccount> through the real @seed-hypermedia/client. It only sees blobs already indexed on the polled server — a comment becomes visible a few seconds after it was authored (propagation + indexing lag).

2Derive a stable key activity-triggers.ts · activityEventKey

Each event gets a key from its feedEventId: a comment event is blob-<cid>; its citation twin is mention-<cid>--<target>. The <cid> (the comment version) is identical in both.

3DROP 1 — the staleness watermark activity-monitor.ts 
const backfillAfter = Math.max(previous.lastSuccessAt || 0, startedAt - ACTIVITY_BACKFILL_MS)
const newEvents = fetched.filter((event) => {
  const key = activityEventKey(event)
  if (!key || previous.seenKeys.includes(key)) return false
  const eventAt = activityEventTimeMs(event)         // = the comment's createTime
  return eventAt === null || eventAt >= backfillAfter  // ← the drop
})

lastSuccessAt is set to Date.now() at the end of every successful poll, so in steady state backfillAfter ≈ now − 5s. An event was only "new" if its eventAt ≥ lastSuccessAt. So any event that became visible in the feed more than ~one poll interval after its own timestamp was permanently dropped (and recorded in seenKeys, so never reconsidered). The flaw: eventAt is the comment's create time, but what we actually care about is observe time — when the event first became visible to us. The feed carries no observe/index time (only create time), so the real observe-time signal is seenKeys itself + the feed's observe-ordered page walk.

Fix (two parts): (1) Request observed-time order. ListEvents defaults to claimed (create) time order, so a late-propagating comment sits buried at its old position and the top-N page walk never fetches it. The monitor now sends order: 'observed' so newly-arrived blobs surface at the top when they propagate. (Plumbed through the client schema + web ListEvents handler.) (2) Decide newness by observation, not create time — an event is "new" simply when its key isn't in seenKeys: fetched.filter(e => key && !seenKeys.includes(key)). The watermark advances per event (immediately after each firing is durably recorded), so a restart resumes from the last handled event. trigger_firings UNIQUE dedups; the create-time bound survives only for the cold-start first poll.

This was the primary reason reply mentions were missed — see the trace below.

4Match against each enabled trigger api-service.ts · processActivityEvent → activity-triggers.ts · matchesSingleMention

Events that survive DROP 1 are checked against the polled account's triggers. A user-mention trigger matches structurally — it does not care whether the comment is a top-level comment or a reply:

if (type === 'comment') {
  // true if any block embeds a mention link resolving to the account
  return commentMentionsAccount(event.comment, mentionedAccount)
}
if (type === 'citation') {
  // BEFORE FIX: if (citationType === 'c') return false   ← DROP 2
  return mentionedAccountFromIdRecord(event.target.id) === mentionedAccount
}
5DROP 2 — comment-sourced citation suppression (now removed) activity-triggers.ts · matchesSingleMention

The citation twin of a comment carries citationType: 'c'. The old code returned false for it, trusting the comment event to fire instead. But when DROP 1 had already discarded the comment event, the citation was the only sibling left — and it was thrown away here. Net: nothing fired.

6Dedup the two siblings & fire once activity-triggers.ts · activityFiringKey → api-service.ts 
// the citation's mention-<cid>--<target> collapses onto the comment's blob-<cid>
const firingKey = activityFiringKey(event)   // both siblings → "blob-<cid>"
INSERT OR IGNORE INTO trigger_firings (... activity_key = firingKey ...)
// UNIQUE(account_id, trigger_id, activity_key) ⇒ exactly one firing ⇒ one session

Whichever sibling reaches this point first creates the session; the other is a no-op insert. Both resolved siblings carry the full comment body (loadCitationEvent fetches it for 'c' citations), so context is identical either way.

Why the reply mentions were missed — the live trace

Three comments in one thread, all mentioning the agent's account. Only the thread root fired.

The thread root won the race only because its comment event happened to surface within one poll interval of its timestamp. Both replies had >5s indexing lag, so DROP 1 discarded their comment events on every poll. The "can you hear me?" reply additionally had a fresh citation that reached the matcher — but DROP 2 suppressed it.

Conclusion: Fix A (committed) resolves the originally-reported "can you hear me?" miss, verified end-to-end. But the broader "reply mentions don't fire" problem is fundamentally DROP 1. Fix B addresses the root: a feed event's freshness is decided by whether we've already observed it (seenKeys), not by its create timestamp — so a comment that propagates arbitrarily late still fires when it first appears to us (extreme propagation lag is expected and should trigger anyway; the policy may become configurable later). A + B compose cleanly: B lets the late comment event through; A's shared-CID firing key prevents the citation twin from double-firing.
