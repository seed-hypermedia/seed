1. Why this exists: the 2026-07-09 outage

On 2026-07-09, hyper.media was unreachable from ~14:05 to ~14:20 UTC (recovered by manual container restarts). Investigation on the production server found the acute trigger was web_gateway wedging after a burst of ~1,650 failed daemon RPCs, but the standing condition that made it fragile was this:

  • Internet vulnerability scanners probe hosted sites with junk paths — phpinfo.php, wp-admin/*, .env, etc. Measured rate: ~1,957 hits/hour, constant background noise.
  • Every such request missed the local database, and the web app reacted by launching a recursive P2P network discovery in the daemon — each attempt burning 5–60 seconds of daemon time dialing peers before failing.
  • Result: daemon_gateway chronically pegged at ~320–360% CPU on a 4-core box, every legitimate RPC slow, and the web tier one burst away from wedging.

The amplification factor is the whole story: one free HTTP GET from a bot cost the server tens of seconds of P2P work. Full incident analysis: Incident: hyper.media outage on 2026-07-09.

2. The amplification mechanism (before)

Before — any HTTP client could start discovery
scanner ──GET /wp-admin/phpinfo.php──▶ web_gateway (SSR loader)
                                          │  resource not found locally
                                          ▼
                                   pokeDiscovery()  ◀── server-side, unconditional
                                          │
                                          ▼
                        daemon: DiscoverEntity(recursive: descendants)
                        ├─ dial peers, walk DHT ……… 5–60 s of CPU/network
                        └─ outcome=error (doc never existed)

  cost to scanner:  1 TCP request, ~0 s
  cost to server:   5–60 s of daemon work  ×  ~33 junk requests/minute, forever

The key defect: the decision to spend expensive P2P resources was made by the server on behalf of any client, with no proof the client was a human with a browser. Bots never see a page render; they fire one GET and move on — but the daemon paid full price every time.

3. The fix, in detail

The design insight (Eric's): make executing our JavaScript the admission ticket for discovery. Vulnerability scanners overwhelmingly do not run JS. Real browsers always do. So: SSR must never start discovery — it should render a fast, cheap shim page whose client-side JS is what actually kicks off the search.

After — only JS execution can start discovery
scanner ──GET /junk.php──▶ web_gateway (SSR loader)
                              │ resource not found locally
                              ▼
                    throw HMDiscoveryPendingError            ◀── NO daemon call
                              ▼
                    404 + shim HTML ("Looking for this document…")
                    scanner sees 404, moves on.  Server cost: one cheap render.

browser ──GET /hm/<uid>/<path>──▶ same 404 + shim HTML
                              │ shim JS runs in the browser
                              ▼
              poll GET /api/DiscoveryStatus?uid=…&path=…     ◀── THE gate
                              ▼
              daemon: DiscoverEntity (starts task if none)   ◀── first poll starts it
                              │ pending → pending → found
                              ▼
              window.location.reload() → real document (200)

3.1 The SSR change — frontend/apps/web/app/loaders.ts

The gateway branch of loadResourceWithDiscovery previously called pokeDiscovery() during SSR — non-blocking, but it started the daemon's discovery task for every not-found request. Now it throws immediately:

if (WEB_IS_GATEWAY) {
  // Never start discovery from SSR. Only clients that execute JS may
  // trigger it: the shim page's first /api/DiscoveryStatus poll is
  // what starts the daemon's discovery task. Bots and vulnerability
  // scanners probe large numbers of nonexistent paths without running
  // JS, and a server-side poke here would queue an expensive recursive
  // P2P discovery task for every probe.
  throw new HMDiscoveryPendingError()
}

The shim response also carries HTTP 404 instead of a soft 200 — scanners and crawlers record "not found" (many scanners probe harder when they see 200s), while browsers happily render 404 HTML and run its JS. When discovery succeeds, the reload returns a genuine 200.

3.2 The trigger endpoint — frontend/packages/shared/src/api-discovery-status.ts

The polling endpoint lives in the main typed API router (APIQueries in @shm/shared) as a first-class DiscoveryStatus query — zod-validated input/output, listed in /api/schema, callable from the SDK, and shared by web and desktop rather than living in a one-off Remix route. The crucial property is inherited from the daemon's DiscoverEntity semantics:

/** …The daemon's discoverEntity call returns immediately with the current
    task state (STARTING A TASK IF NONE EXISTS)… This is the only way
    discovery gets started for gateway page loads. */
const resp = await grpcClient.entities.discoverEntity({
  id: discoveryUrl({uid, path, recursion: 'descendants'}),
  version: version || undefined,
})

So the flow needs no extra bookkeeping: the shim's first poll is simultaneously the trigger and the status check. Repeated polls are idempotent check-ins, and the endpoint never holds a request open.

3.3 The shim — frontend/apps/web/app/routes/$.tsx (DiscoveryPendingPage)

  • Rendered server-side in one cheap pass (no daemon RPCs) with a spinner and "Looking for this document…".
  • Client JS polls /api/DiscoveryStatus every 2 s, up to 60 polls (~2 min), then shows "Document Not Found" with a Try Again button.
  • On state: 'found' → window.location.reload() → the document is now local, so SSR serves it normally.

Test 1 — the scanner can no longer spend our money

  1. Plain fetch() (no JS — exactly what a scanner is) requests /hm/<uid>/discovery-e2e on the gateway.
  2. Assert: response is HTTP 404 and contains the shim page ("Looking for this document").
  3. Wait 15 seconds — ample time for any discovery to complete on a two-node LAN testnet (Test 2 shows discovery completes in ~5 s here).
  4. Assert: the document still does not exist on the gateway daemon (/api/Resource → not-found), and a second no-JS fetch still gets the 404 shim.

What this proves: the doc was one hop away, peers were connected, discovery would have succeeded in seconds — yet no discovery happened, because nothing ran the JS. If SSR still poked discovery, the doc would have synced and this assertion would fail (we verified that failure mode empirically: an earlier test iteration where the daemon subscribed to the content account showed the doc syncing within seconds of connection).

Test 2 — the real user is unharmed

  1. Playwright Chromium loads the same URL. Assert: shim visible.
  2. The page's own JS polls /api/DiscoveryStatus. Observed: pending → pending → found (~3 polls, ≈5 s), each poll logged by the test with its response body.
  3. The shim reloads itself. Assert: the unique marker text (DISCOVERY-E2E-MARKER-…) is rendered — content that could only have arrived via P2P discovery from daemon A.
  4. Assert: at least one /api/DiscoveryStatus request was observed (right mechanism, not a side channel), and the doc now exists on the gateway daemon.

5. Why the evidence is conclusive

DoubtHow the test eliminates it
"Maybe the doc didn't sync in Test 1 because the peers weren't connected." The connection is established and asserted before Test 1 (/hm/api/register → networking.connect returns success), and Test 2 proves the same link moves the doc in ~5 s. The 15 s negative window is 3× that.
"Maybe discovery was triggered but just slow." A second no-JS fetch after the wait still returns the 404 shim, and /api/Resource on the gateway still reports not-found. A triggered discovery on this topology completes in seconds (measured in Test 2).
"Maybe the browser got the content some other way (subscription, bulk sync)." The gateway's registered home account is a deliberately unrelated, empty account — an earlier test iteration proved that registering the content account causes subscription bulk-sync, so the test isolates that path away. The only remaining transport is discovery, and the test additionally asserts the shim actually called /api/DiscoveryStatus.
"Maybe it works in the test env but not like production." Every layer is the real thing: the actual Go daemon binary (with real libp2p networking), the actual Remix web app in gateway mode (SEED_IS_GATEWAY=true), real signed blobs published through the real API, a real browser. Only scale and network distance are smaller.
"Maybe the fix breaks real users." Test 2 is a real user: from a cold gateway, first paint of the shim → discovered document in ~5–8 s, with zero manual steps.

7. Issues found & fixed along the way

The test run only passed on the sixth iteration — each failure was a real discovery. In order:

7.1 OPEN — backend bug Hyphenated testnet names break P2P entirely

backend/hmnet/hmnet.go:740 — hmProtocolPattern = ^\/hypermedia\/\d\.\d\.\d(-\w+)?$. The testnet suffix only admits \w, so a testnet named cli-test-123-abc produces protocol ID /hypermedia/x.y.z-cli-test-123-abc, which fails the regex — peers reject each other as "not a Hypermedia peer". The CLI test helper generates exactly such hyphenated names, meaning no multi-daemon P2P test could ever have worked with the stock helper. Recommend widening the regex (e.g. (-[\w-]+)?) or validating the flag. The e2e works around it with hyphen-free names.

7.2 FIXED Test infra leaked every daemon and web server it ever started

frontend/apps/cli/src/test/setup.ts spawns the daemon via sh -c "go run …" and killed only the shell — go run's compiled child survived every cleanup (we found daemons from old test runs still alive). Same pattern for web servers, with a nastier twist: leaked vite processes watch the shared apps/web/.env, so a "dead" server from run N restarts itself with run N+1's config mid-test. Leaked stdio pipes also kept the test process alive 47 minutes after all tests had passed. Fixed with detached: true + process-group kills (kill(-pid)), hard timeouts on every fetch, and an explicit exit. This fix benefits the pre-existing CLI fixture tests too.

7.3 FIXED (in test design) Subscriptions bulk-sync content without discovery

frontend/apps/web/app/site-config.server.ts (applyConfigSubscriptions) recursively subscribes the daemon to the site's registeredAccountUid. When the test's gateway config pointed at the fixture account, the doc arrived via subscription sync seconds after connecting — a false "discovery leak" that was actually correct subscription behavior. The test now registers an unrelated empty home account, which is also what makes its negative assertion airtight.

7.4 OPEN — same bug class in prod Non-gateway SSR still blocks on discovery

The e2e's readiness probe accidentally reproduced the production outage in miniature: probing / on a not-yet-seeded non-gateway server made SSR block inside discoverDocument → tryUntilSuccess (the exact stack from the prod incident), stalling every 5-second probe until the 120 s deadline. The gateway path is now fixed, but self-hosted sites (non-gateway mode) still block SSR requests on discovery — the same wedge mechanism remains available there and deserves the same shim treatment. The test works around it by probing /hm/api/config instead of /.

7.5 FIXED Gateway requires a registered home account; vite silently drifts ports

Two smaller finds: (a) the route loader returns an "unregistered" page unless config.json has a registeredAccountUid, even in gateway mode — the test config now mirrors prod; (b) vite auto-increments to a free port when the requested one is busy, so the harness now parses the actual Local: URL from vite's output instead of trusting the requested port.

8. Limitations & follow-ups

  • Not deployed yet. All of this is uncommitted working-tree code (on top of the pre-existing async-shim WIP). The production image predates even the shim and still uses fully blocking SSR discovery. Until deploy, the traefik-level scanner-path blocking from the incident report remains the stopgap.
  • Headless-browser scanners can still trigger discovery. If that ever matters, add per-IP rate limiting on /api/DiscoveryStatus — a single choke point now exists, which is itself a win.
  • Daemon-side hardening still worthwhile: negative-result caching for failed discoveries and a bounded concurrent-discovery budget would protect against any triggered-discovery flood, JS or not.
  • Non-gateway mode (7.4) should eventually get the same shim pattern.
  • The ACME retry storm for stale custom domains is an unrelated ongoing issue from the incident and is untouched by this fix.
